## Hard rules
- 2 spaces for indentation
- All code must work for macOS, Linux and Windows
- Must write idiomatic Go at any cost overriding any other rules here
- In the event of CLI code changes, you must keep the `docs/public` folder up-to-date
- Every on-disk path component (directory name, filename, extension) referenced by Go source MUST be a named constant in `constants.go`. Compose paths from those constants via `filepath.Join` or string concatenation; never write a string literal like `".x-x"`, `"_config.lock"`, or `".x-plans/foo.yaml"` inline in a function body. Adding or renaming a path = edit `constants.go` first, then update call sites. The same rule applies to non-Go source that asserts on those paths (`scripts/e2e_test.sh`, future shell/PowerShell harnesses): mirror the relevant `constants.go` values once at the top of the file as named variables and use `${VAR}` interpolation throughout — never inline a path literal like `".x-plans/_config.lock"` next to an assertion. Drift between the mirrored block and `constants.go` MUST be caught by a unit test that imports the Go constants and re-reads the mirror.
- Never run bare `go build` (or `go build .` / `go build ./...`) at the repo root — it drops the compiled binary at `./x-x`, polluting the working tree. Use `task build` for release artifacts (writes to `./bin/`). For compile-only verification, use `go vet ./...` or `go build -o /dev/null ./...`.
- Any change to a CLI source file (`*.go` at the repo root) MUST land with accompanying tests, and those tests MUST run from `.github/workflows/test.yml`. Add unit coverage to the matching `*_test.go` file and, when the change touches a user-visible CLI surface (a subcommand, a flag, an on-disk side effect, or stdout/stderr contract), extend `scripts/e2e_test.sh` so the workflow exercises the new behavior inside its Linux container. A PR that adds or modifies CLI behavior without updating both is not mergeable.
- Windows e2e parity: whenever you touch `scripts/e2e_test.sh`, evaluate the same change against `scripts/e2e_test.ps1`. Behavior-neutral assertions (anything not depending on POSIX file modes, symlink semantics, drive letters, reserved Windows filenames, PATH registry writes, or shell-runtime specifics) MUST land in both files in the same change. Platform-specific cases may live in only the relevant file. The bash e2e runs on every PR via `.github/workflows/test.yml`; the PowerShell e2e also runs on every PR via `.github/workflows/windows-cli.yml` (and on push to `main`), with a `workflow_dispatch` re-run path for the loud, debug-logged form. A PR may skip the PS1 update only with an explicit comment noting why (e.g. "POSIX-only assertion: no Windows analogue").
- Every interactive prompt MUST have a fully non-interactive equivalent that drives the same code path via flags (any number of them — one per choice the prompt would collect). The non-interactive form must take a single shot of arguments and exit without ever reading from stdin. Both forms must be covered in `scripts/e2e_test.sh`: at minimum one case piping input into the prompt and one case passing the equivalent flags. Rationale: CI, scripted installs, and headless agents cannot answer prompts, so every interactive scenario that ships without a flag-driven twin is a blocker for automation. Example: `x-x init` prompts for scope AND accepts `--scope project|user` — when a new prompt question is added, a matching flag must be added in the same change.
- `~/.x-x/agents/` is owned exclusively by the `x-x` binary's embedded FS. It is materialized on first run (`ensureBundledAgents`) and overwritten wholesale every 24h by `maybeNotifyUpdate` → `writeBundledAgents(true)`. Users MUST NOT hand-edit anything under this tree — local changes are destroyed on the next refresh. Conversely, no code path other than `writeBundledAgents` (and its callers `ensureBundledAgents` / `maybeNotifyUpdate`) may write into `~/.x-x/agents/`.
- Project-level skill trees (`.claude/`, `.agents/`, `.codex/` inside a user's project) are written by `x-x init` and are off-limits to `x-x` thereafter. No subcommand, update check, or background path may mutate, refresh, or delete files under these directories after init — they are the user's to edit, version-control, and diverge from the embed. The 24h refresh in `maybeNotifyUpdate` is scoped strictly to `~/.x-x/agents/` and MUST NOT extend to project trees. Any future feature that re-touches project skills must be an explicit, user-invoked command, never an opportunistic background write.
- Every commit MUST be both DCO-signed-off AND cryptographically signed before being pushed. DCO sign-off is the `Signed-off-by: <Name> <email>` trailer matching the commit's author email — pass `-s` to every `git commit` (or set `git config --local format.signoff true` once per clone). Cryptographic signing produces the green "Verified" badge on GitHub — requires `git config --local commit.gpgsign true`, `git config --local gpg.format ssh` (or `gpg`), and `git config --local user.signingkey <path-or-id>` pointing at an SSH/GPG key registered to the author's GitHub account under **Signing Key** type at https://github.com/settings/keys. Checking `git config --get commit.gpgsign` is NOT sufficient — config can say `true` while signing silently no-ops if the key file is missing, the agent isn't loaded, or the format is misconfigured. Verify the previous commit actually signed: `git log -1 --format='%G?'` MUST return `G`, `U`, or `B` (signed); `N` means no signature was attached and the push will land unverified. If `%G?` returns `N`, STOP and ask the user to complete signing setup per `docs/internal/commit-signing.md` rather than pushing an unverified commit. Both DCO and signature checks are branch-protection enforced on `main`; pushing an unverified commit forces a force-push to fix and burns DCO/signature history.
