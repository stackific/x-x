## What this project is

`stax` (Stackific Inc.) is a Go CLI that ships a two-skill agent vocabulary — `/scope` (planner) and `/ship` (executor) — to Claude Code, Codex CLI, and any other agent that follows the SKILL.md open standard. The binary's job is to install those skills into each agent's expected locations and seed a project scaffold (`.stax/`) that the skills write work-item files into.

Two persistent on-disk surfaces matter:

- **`~/.stax/agents/`** — the bundled embed, materialized on first run, refreshed wholesale every hourly. The binary owns it; users MUST NOT hand-edit.
- **`<project>/.stax/`** — the per-project scaffold (`_config.lock` pins, `_data_systems.yaml` registry, `*.md` work-item files). `stax init` seeds it; everything after that is the user's, including the work-item files which are first-class repo content.

The single on-disk marker that distinguishes "a stax project" from "any other directory" is `<cwd>/.stax/_config.lock`. Every project-scope CLI subcommand (`stax work-items *`, `stax skills remove --project`) and every Claude/Codex invocation of `/scope` or `/ship` keys off that file. If it is missing, the directory is unusable as a stax project regardless of whether skills are installed.

`--scope` on `stax init` decides ONLY where the bundled SKILLS land:

- `--scope project` → skills under `<cwd>/.claude/skills/`, `<cwd>/.agents/skills/`, …
- `--scope user`    → skills under `~/.claude/skills/`, `~/.agents/skills/`, …

`.stax/_config.lock` is seeded in cwd under **both** scopes. A user-scope install that did not seed cwd would produce skills with nowhere to anchor work items — the project marker check would fail on the very next command. The two responsibilities (skill placement and project-marker scaffold) are orthogonal; treat them that way in every code path and every test assertion.

## Hard rules
- 2 spaces for indentation
- All code must work for macOS, Linux and Windows
- Must write idiomatic Go at any cost overriding any other rules here
- In the event of CLI code changes, you must keep the `docs/public` folder up-to-date
- Every on-disk path component (directory name, filename, extension) referenced by Go source MUST be a named constant in `constants.go`. Compose paths from those constants via `filepath.Join` or string concatenation; never write a string literal like `".stax"`, `"_config.lock"`, or `".stax/foo.yaml"` inline in a function body. Adding or renaming a path = edit `constants.go` first, then update call sites. The same rule applies to non-Go source that asserts on those paths (`scripts/e2e_test.sh`, future shell/PowerShell harnesses): mirror the relevant `constants.go` values once at the top of the file as named variables and use `${VAR}` interpolation throughout — never inline a path literal like `".stax/_config.lock"` next to an assertion. Drift between the mirrored block and `constants.go` MUST be caught by a unit test that imports the Go constants and re-reads the mirror.
- Never run bare `go build` (or `go build .` / `go build ./...`) at the repo root — it drops the compiled binary at `./stax`, polluting the working tree. Use `task build` for release artifacts (writes to `./bin/`). For compile-only verification, use `go vet ./...` or `go build -o /dev/null ./...`.
- Any change to a CLI source file (`*.go` at the repo root) MUST land with accompanying tests, and those tests MUST run from `.github/workflows/test.yml`. Add unit coverage to the matching `*_test.go` file and, when the change touches a user-visible CLI surface (a subcommand, a flag, an on-disk side effect, or stdout/stderr contract), extend `scripts/e2e_test.sh` so the workflow exercises the new behavior inside its Linux container. A PR that adds or modifies CLI behavior without updating both is not mergeable.
- Windows e2e parity: whenever you touch `scripts/e2e_test.sh`, evaluate the same change against `scripts/e2e_test.ps1`. Behavior-neutral assertions (anything not depending on POSIX file modes, symlink semantics, drive letters, reserved Windows filenames, PATH registry writes, or shell-runtime specifics) MUST land in both files in the same change. Platform-specific cases may live in only the relevant file. The bash e2e runs on every PR via `.github/workflows/test.yml`; the PowerShell e2e also runs on every PR via `.github/workflows/windows-cli.yml` (and on push to `main`), with a `workflow_dispatch` re-run path for the loud, debug-logged form. A PR may skip the PS1 update only with an explicit comment noting why (e.g. "POSIX-only assertion: no Windows analogue").
- Every interactive prompt MUST have a fully non-interactive equivalent that drives the same code path via flags (any number of them — one per choice the prompt would collect). The non-interactive form must take a single shot of arguments and exit without ever reading from stdin. Both forms must be covered in `scripts/e2e_test.sh`: at minimum one case piping input into the prompt and one case passing the equivalent flags. Rationale: CI, scripted installs, and headless agents cannot answer prompts, so every interactive scenario that ships without a flag-driven twin is a blocker for automation. Example: `stax init` prompts for scope AND accepts `--scope project|user` — when a new prompt question is added, a matching flag must be added in the same change.
- `~/.stax/agents/` is owned exclusively by the `stax` binary's embedded FS. It is materialized on first run (`ensureBundledAgents`) and overwritten wholesale every hourly by `maybeNotifyUpdate` → `writeBundledAgents(true)`. Users MUST NOT hand-edit anything under this tree — local changes are destroyed on the next refresh. Conversely, no code path other than `writeBundledAgents` (and its callers `ensureBundledAgents` / `maybeNotifyUpdate`) may write into `~/.stax/agents/`.
- Project-level skill trees (`.claude/`, `.agents/`, `.codex/` inside a user's project) are written by `stax init` and are off-limits to `stax` thereafter. No subcommand, update check, or background path may mutate, refresh, or delete files under these directories after init — they are the user's to edit, version-control, and diverge from the embed. The hourly refresh in `maybeNotifyUpdate` is scoped strictly to `~/.stax/agents/` and MUST NOT extend to project trees. Any future feature that re-touches project skills must be an explicit, user-invoked command, never an opportunistic background write.
- `stax init` MUST seed `<cwd>/.stax/` (`_config.lock` + `_data_systems.yaml`) under **every** scope, including `--scope user`. `--scope` controls only where the bundled SKILLS land (project tree vs `$HOME`); the `.stax/_config.lock` marker is what makes cwd usable with `/scope`, `/ship`, and every `stax work-items *` subcommand, and it has nothing to do with where skills live. Do NOT condition `writeWorkItemsScaffold` on scope — that produced skills with nowhere to anchor work items and failed the "not a stax project" check on every subsequent command. Tests that assert the *absence* of `.stax/` after `--scope user` are wrong and lock in this bug; tests MUST assert presence of `_config.lock` and `_data_systems.yaml` in cwd under both scopes.
- Every commit MUST be both DCO-signed-off AND cryptographically signed before being pushed. DCO sign-off is the `Signed-off-by: <Name> <email>` trailer matching the commit's author email — pass `-s` to every `git commit` (or set `git config --local format.signoff true` once per clone). Cryptographic signing produces the green "Verified" badge on GitHub — requires `git config --local commit.gpgsign true`, `git config --local gpg.format ssh` (or `gpg`), and `git config --local user.signingkey <path-or-id>` pointing at an SSH/GPG key registered to the author's GitHub account under **Signing Key** type at https://github.com/settings/keys. Checking `git config --get commit.gpgsign` is NOT sufficient — config can say `true` while signing silently no-ops if the key file is missing, the agent isn't loaded, or the format is misconfigured. Verify the previous commit actually signed: `git log -1 --format='%G?'` MUST return `G`, `U`, or `B` (signed); `N` means no signature was attached and the push will land unverified. If `%G?` returns `N`, STOP and ask the user to complete signing setup per `docs/internal/commit-signing.md` rather than pushing an unverified commit. Both DCO and signature checks are branch-protection enforced on `main`; pushing an unverified commit forces a force-push to fix and burns DCO/signature history.

## Frontend stack and conventions

The web UI under `frontend/` is a Vite multi-page app (MPA), not an SPA. Routing is browser-native — one `.html` file per route, one shared JS bundle, no client-side router. The Go binary embeds the built `frontend/dist/` via `//go:embed all:frontend/dist` and serves it at `localhost:7829`; `server.go` adds clean-URL fallback (`/systems` → `systems.html`) and a branded `404.html` on misses.

**Stack:**
- **Vite** with `vite-plugin-handlebars` for build-time block partials. `frontend/partials/layout.html` is the shell; `header`, `mobile-nav`, `footer` are nested partials. Each page calls `{{#> layout title="..." page="..." active="..."}}...{{/layout}}`.
- **TypeScript** + **BeerCSS** (Material Design 3 utility classes + Material Symbols icons) + **material-dynamic-colors** for the M3 palette. **Geist Variable** is loaded by a single `@font-face` rule in `src/styles/app.scss` pointing at jsdelivr's fontsource CDN — no woff2 embedded in the build output.
- **Biome** for lint + format, **Knip** for deadcode, **Terser** for minification. `npm run check` runs typecheck + Biome + Knip; the lefthook `frontend-check` pre-push hook runs the same.
- **goldmark** (server side, server.go via `markdownRenderer`) renders work-item markdown to HTML before crossing the wire — GFM extension, headings forced to `<h6>` by the `forceH6Headings` AST transformer, raw HTML escaped (no `WithUnsafe`). The frontend `innerHTML`s only that sanitized output; user-typed markdown never reaches the page directly.

### Frontend hard rules
- **No HTML-in-JS.** Page markup lives in `.html` files. Dynamic content is rendered by cloning a `<template>` element and writing `textContent` / `href` into `data-slot="..."` placeholders (see `src/shared/dom.ts` for `$` and `tpl`). Building DOM via template-literal HTML strings or `innerHTML = "<div>..."` is forbidden — it defeats the no-XSS contract and pulls layout into TS. The single exception is `#work-item-body` on `/work-item?id=`, where the server returns goldmark-rendered, raw-HTML-escaped output.
- **One JS entry, one CSS file.** `src/bundle.ts` is the only JS entry across every page; `dist/` ships `bundle.js` + `bundle.css`. A post-build `renameBundle` Vite plugin restores the names after Rollup's auto-suffixed `bundleN.js` dedup. Adding a page means four coordinated edits in the same change: (a) new `<page>.html` calling the layout partial, (b) entry in the `pages` array in `vite.config.ts`, (c) per-page init function in `src/pages/<page>.ts`, (d) wiring in the dispatcher in `src/bundle.ts`.
- **Dispatch by `data-page` and `data-active`.** `<body data-page="<name>">` (set per page via the layout's block-helper context) drives the per-page init dispatcher in `bundle.ts`. Nav highlight uses `<body data-active="<key>">` for parity (a `/work-item?id=…` page sets `active="work-items"` so the sidebar still highlights the list). Both attributes are written by the HTML — never inferred from `location.pathname` in TS.
- **Color cues go through `src/shared/status.ts`.** `applyStatusClass(el, status)` tints status nodes per lifecycle (`superseded` → tertiary, `deprecated` → error) and rows whose work item has at least one `- [ ]` task get `primary-text` on the flag icon. Adding a new semantic cue means extending the helper, not inlining a color class per call site. Use BeerCSS color tokens (`primary` / `secondary` / `tertiary` / `error`, with `-text` / `-container` variants) rather than custom CSS variables — the M3 dynamic palette already covers the conventional roles.
- **Relative time goes through `src/shared/relative-time.ts`** (`relativeTime(iso)`). Don't reach for `toLocaleDateString` per page; the helper handles "just now" / "N minutes ago" / "N months M days ago" uniformly.
- **Shared modules under `src/shared/`** — `api` (typed fetch wrapper), `dom` (`$` / `tpl` helpers), `qs` (URL query string getter), `theme`, `sidebar`, `nav`, `relative-time`, `status`. New cross-page utilities go here, not duplicated per page. Importing across `src/pages/*.ts` is forbidden — pages are dispatcher leaves and stay independent.
- **API shape conventions (must mirror `server.go`):** list responses wrap the array as `{things: [...]}` (`{systems: [...]}`, `{workItems: [...]}`) for forward compat with pagination cursors; empty results are serialized as `[]`, never `null`; cross-references between entities (e.g. `supersededBy`, `supersedes`) use `{slug, title}` (or `{id, name}`) pairs so the UI can render human text without a per-row round-trip. Adding a field is a breaking change on the wire only if it's required — optional fields land as `omitempty` in Go and `field?: T` in TS.
- **Dev proxy is the single source of truth for the backend URL.** `frontend/vite.config.ts` proxies `/api → http://localhost:7829` (also a constant: `serverListenAddr` / `serverDisplayURL` in `constants.go`). Page TS calls `/api/...` directly; don't hardcode the host:port anywhere in TS or HTML.
- **HTML / CSS authoring lives in `frontend/`, not in Go.** The Go server never templates HTML — it serves the embedded static `dist/` tree plus JSON. Any "new page" work is a frontend change; backend changes ship new JSON shapes the page can render against.
